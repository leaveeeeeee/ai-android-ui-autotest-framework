# ADR 0002: 多设备执行先定义 DevicePool 路线，再决定何时实现

## 状态

已采纳

## 背景

当前框架的真实执行语义是：

- 单 `serial`
- 单设备
- `device` 用例禁止与 `pytest-xdist` 并行

这不是一个简单的线程安全开关问题，而是运行时上下文、报告路径、步骤采集、设备恢复语义都默认围绕“当前只有一台设备”来设计。

## 决策

本轮不直接实现多设备执行，而是先明确未来路线：

- 公开配置草案：`device.serials`
- 运行时抽象：`DevicePool`
- worker 语义：一个 worker 在同一时刻只 lease 一台设备
- 生命周期：worker 启动时申请设备，测试结束或崩溃时归还
- 失败回收：设备恢复失败时标记为 dirty，避免继续分配
- 报告隔离：`run_id / case_name / worker / serial` 共同参与产物命名

## 原因

- 当前最紧急的问题仍是步骤 API、可测试性和文档一致性
- 如果在这些基础设施还没有稳定前直接放开多设备执行，故障面会成倍扩大
- 先写 ADR，可以让后续实现有清晰边界，避免“先加参数，再补设计”

## 当前结论

- `device.serial` 继续作为当前唯一正式支持的配置
- `device.serials` 只进入设计草案，不进入运行时代码
- `device + xdist` 禁止策略继续保留

## 后续进入实现阶段前的前置条件

- `DriverAdapter` / runtime context / 报告路径具备 worker 级隔离能力
- 失败恢复流程能区分“当前设备失败”和“池内其他设备可继续执行”
- 真机报告产物能稳定带出 `serial` 和 `worker` 维度
